# 任务执行说明书：项目审查与安全优化（白名单式）

## 【角色定义】

你是一个专门负责对 Phaser/Vite 塔防游戏做 **结构化审查** 与 **白名单式安全优化** 的助手。你只按本说明书列出的检查项与修改项工作；任何不在白名单上的 “优化” 都不允许执行，必须只列入审查报告并标 `[待确认]`。

## 【任务目标】

按本说明书的 14 项检查清单逐项审查仓库 `c:\MyGame`，输出一份审查报告 `doc/规划/项目审查报告.md`，并把白名单中标记为 `自动应用` 的小步改动直接落地到代码。**不要做大重构、不要拆分文件、不要修改玩法、不要修改 `src/scenes/GameScene.js` 中除导入语句外的任何业务逻辑**。

最终交付物：

1. `doc/规划/项目审查报告.md`：固定结构的报告。
2. 白名单内 “自动应用” 类改动若干（详见执行步骤）。
3. `npm run build` 退出码 0。

## 【执行步骤】

### A 阶段：先决条件

1. 检查 `doc/规划/` 下是否已存在 `项目审查报告.md`：
   - 若存在：把旧文件**重命名**为 `项目审查报告.<时间戳>.md`（时间戳格式 `YYYYMMDD-HHmm`，使用本机当前时间），不要删除。
   - 若不存在：跳过本步。

2. 在仓库根列出当前文件结构（不要改动），把以下命令的输出原样写入审查报告草稿区域 “0. 仓库快照”：

   ```powershell
   git rev-parse HEAD
   git status --short
   ```

   若不在 git 仓库中：写入 `[非 git 仓库]`。

### B 阶段：14 项审查清单（逐项执行 → 在报告中记录）

按下表逐项审查。每项必须输出 4 个字段：**检查项 / 现状 / 结论 / 处置**。

“处置” 取值仅以下三种：

- `自动应用：<简述改动>` —— 必须在 C 阶段执行白名单改动。
- `[待确认]：<原因>` —— 不改，仅列入报告。
- `通过` —— 现状已合格。

| # | 检查项 | 通过条件 |
|---:|---|---|
| 1 | `package.json` 含 `"type": "module"` | 字段存在且值为 `"module"` |

| 14 | `doc/规划/` 下是否存在多份未执行的任务说明书 | 列出文件名清单（仅记录） |

### C 阶段：白名单改动（自动应用）

仅以下 5 类改动允许直接落地。每类都给出 **触发条件** 与 **完整代码**。其它任何修改都禁止。

#### C-1 补全 `.gitignore`

触发条件：第 3 项缺任意条目。

操作：在 `.gitignore`（不存在则新建）末尾追加缺失行（每行一个，不要重复已有行）：

```
node_modules
dist
.vite
.DS_Store
*.log
```

#### C-2 新增 `jsconfig.json`

触发条件：第 4 项不通过。

操作：在仓库根新建 `jsconfig.json`，内容**逐字**：

```json
{
  "compilerOptions": {
    "module": "ESNext",
    "target": "ES2020",
    "moduleResolution": "Bundler",
    "checkJs": false,
    "allowJs": true,
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    }
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist", ".vite"]
}
```

不要修改 `vite` 别名。`paths` 仅供编辑器索引，不改变运行时。

#### C-3 抽出 `src/utils/storage.js`

触发条件：第 9 项命中 ≥ 2 处。

操作：

1. 新建 `src/utils/storage.js`，内容**逐字**：

   ```js
   const PREFIX = "crown-outpost-";

   export const STORAGE_KEYS = {
     bestWave: PREFIX + "best-wave",
     easterEggs: PREFIX + "easter-eggs",
     save: PREFIX + "save-v1",
   };

   export function readJSON(key, fallback = null) {
     try {
       const raw = localStorage.getItem(key);
       return raw == null ? fallback : JSON.parse(raw);
     } catch {
       return fallback;
     }
   }

   export function writeJSON(key, value) {
     try {
       localStorage.setItem(key, JSON.stringify(value));
     } catch {}
   }

   export function readString(key, fallback = "") {
     const v = localStorage.getItem(key);
     return v == null ? fallback : v;
   }

   export function writeString(key, value) {
     try {
       localStorage.setItem(key, String(value));
     } catch {}
   }

   export function removeKey(key) {
     try {
       localStorage.removeItem(key);
     } catch {}
   }
   ```

2. **不要** 在本任务里改写 `GameScene.js` 已有的 `localStorage.*` 调用。仅仅引入此工具文件，作为后续迁移的基础。在审查报告里把 `GameScene.js` 中现有的 `localStorage` 调用位置（行号）列出 + 标 `[待确认] 后续任务迁移到 STORAGE_KEYS`。

#### C-4 写入 / 补全 `README.md`

触发条件：第 11 项不通过（不存在或首行为空）。

操作：在仓库根创建 / 覆盖 `README.md`（**仅当不存在时创建**；存在但首行为空时，仅在文件最前插入下列模板，再保留旧内容）：

#### C-5 `package.json` 补 `engines`

触发条件：第 2 项不通过。

操作：在 `package.json` 顶层添加 `"engines": { "node": ">=18" }`。**不要** 修改其它字段、不要重排字段顺序之外的内容。

> 若 `package.json` 含 trailing comma / 注释等非法 JSON：跳过本项，标 `[待确认] package.json 非纯 JSON，需要手工调整`。

- 项目较大且 `GameScene.js` 超过 50KB：本任务**只记录、不拆分**。拆分留给后续 `不动标：` 触发的专项任务。

- 若同一项检查同时满足多个白名单条目：依顺序仅执行第一个匹配的白名单项，其它仅记录。
- 不确定是否需要执行某个白名单条目时：**不执行**，标 `[待确认]` 并解释原因。
- 不确定文件编码 / 换行符时：保持原样，**不要** 整体重写文件以 “统一格式”。
